# EazyClass

EazyClass — серверная часть системы для работы с учебным расписанием.
Проект реализован на Django и предоставляет REST API, административный
интерфейс, механизм синхронизации расписания из внешних источников,
пользовательские подписки и рассылку уведомлений.

## Возможности

- хранение учебного расписания: факультеты, группы, преподаватели, аудитории,
  дисциплины, периоды и занятия;
- REST API на базе Django REST Framework;
- поддержка формата JSON:API для ресурсных endpoint'ов;
- административный интерфейс Django для сопровождения данных без прямого
  доступа к базе данных;
- подписки пользователей на группу или преподавателя;
- синхронизация расписания из внешнего источника;
- отправка уведомлений об изменении расписания и предстоящих занятиях;
- фоновые задачи на Celery;
- управление периодическими задачами через `django-celery-beat`;
- документация API через Swagger и Redoc.

## Структура Проекта

```text
eazyclass/        настройки Django-проекта, URL routing, Celery
scheduler/       основное приложение: модели, API, админка, синхронизация, уведомления
scrapy_app/      Scrapy-интеграция для сбора данных расписания
utils/           общие вспомогательные компоненты
nginx/           конфигурация nginx для production
certbot/         файлы для работы с SSL-сертификатами
rclone/          конфигурация удалённого хранилища для бэкапов
data/            локальные входные данные для фоновых задач
```

Основная логика сосредоточена в приложении `scheduler`. Оно содержит доменную
модель расписания, API-слой, настройки административной панели, механизм
синхронизации, подсистему уведомлений и фоновые задачи.

## Переменные Окружения

Настройки проекта передаются через переменные окружения. В качестве шаблона
используется файл `env.example`.

Для локальной разработки:

```bash
cp env.example .env.dev
```

В PowerShell:

```powershell
Copy-Item env.example .env.dev
```

Для production-окружения создаётся файл `.env` на основе того же шаблона.

Основные группы настроек:

- Django: `DEBUG`, `SECRET_KEY`, `ALLOWED_HOSTS`, `CSRF_TRUSTED_ORIGINS`;
- PostgreSQL: `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD`,
  `POSTGRES_HOST`, `POSTGRES_PORT`;
- Redis и Celery: `REDIS_*`, `CELERY_BROKER_URL`, `CELERY_RESULT_BACKEND`;
- авторизация: `JWT_SIGNING_KEY`, `TELEGRAM_HMAC_SECRET`, `VK_HMAC_SECRET`;
- внешний источник расписания: `BASE_SCRAPING_URL`;
- Telegram: `TELEGRAM_BOT_TOKEN`, `TELEGRAM_BOT_USERNAME`;
- Flower: `FLOWER_USERNAME`, `FLOWER_PASSWORD`.

Реальные токены, пароли и секретные ключи нельзя коммитить в репозиторий.

## Запуск через Docker (development)

Собрать и запустить сервисы:

```bash
docker compose -f docker-compose.dev.yml up --build
```

Применить миграции:

```bash
docker compose -f docker-compose.dev.yml exec django python manage.py migrate
```

Создать администратора:

```bash
docker compose -f docker-compose.dev.yml exec django python manage.py createsuperuser
```

Заполнить стандартный шаблон времени занятий:

```bash
docker compose -f docker-compose.dev.yml exec django python manage.py filltemplate
```


## Локальный запуск без Docker

Рекомендуемый способ запуска — Docker, так как проект использует PostgreSQL,
Redis, Celery и Scrapy. Для запуска без Docker необходимо заранее поднять
PostgreSQL и Redis и корректно заполнить `.env.dev`.

```bash
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
python manage.py migrate
python manage.py filltemplate
python manage.py runserver
```

Для Linux/macOS:

```bash
source venv/bin/activate
```

## API

API доступно по префиксу:

```text
/api/v1/
```

Основные группы endpoint'ов:

- `lessons` — занятия расписания;
- `groups` — учебные группы;
- `teachers` — преподаватели;
- `faculties` — факультеты;
- `classrooms` — аудитории;
- `users` — профиль пользователя;
- `subscriptions` — подписки текущего пользователя;
- `group-subscriptions` — подписки на группы;
- `teacher-subscriptions` — подписки на преподавателей;
- `social-accounts` — социальные аккаунты пользователя.

Также есть endpoint'ы для получения и обновления токенов, выхода из системы,
авторизации через Telegram Web App, генерации deeplink и привязки nonce.

Ресурсные endpoint'ы используют формат JSON:API. Служебные endpoint'ы
авторизации и работы с токенами используют обычный JSON.

## Административный Интерфейс

Django Admin используется как эксплуатационный интерфейс для сопровождения
данных без прямого доступа к базе. Через него можно управлять:

- справочниками расписания;
- занятиями и периодами;
- пользователями и социальными аккаунтами;
- подписками;
- шаблонами времени занятий;
- периодическими задачами Celery Beat.

В админке настроены поиск, фильтры, autocomplete-поля, массовое включение и
отключение записей, inline-редактирование временных интервалов и дополнительная
валидация форм расписания.

## Синхронизация Данных

Механизм синхронизации разделён на два этапа:

1. получение сырых данных из внешнего источника;
2. синхронизация подготовленных данных с базой данных.

Основным источником данных является Scrapy-паук, который проходит по страницам
расписания и собирает данные о занятиях. Для отдельных простых страниц, например
списка групп или преподавателей, используется обычная загрузка HTML и парсинг
без запуска Scrapy.

После подготовки данных общий синхронизатор:

- нормализует поля занятий;
- сопоставляет связанные сущности;
- создаёт отсутствующие справочные объекты;
- сравнивает новое расписание с текущим состоянием базы;
- добавляет новые занятия;
- обновляет изменённые занятия;
- удаляет устаревшие записи;
- формирует сводку изменений.

Такой подход позволяет заменить источник данных, не переписывая основной
алгоритм синхронизации с базой.

## Уведомления

Подсистема уведомлений информирует пользователей об изменениях расписания и
предстоящих занятиях. Она работает поверх модели подписок:

- пользователь подписывается на группу или преподавателя;
- настройки пользователя определяют, какие уведомления он получает;
- социальный аккаунт хранит идентификатор канала доставки, например Telegram
  `chat_id`.

Общий процесс рассылки:

1. определяется событие: изменение расписания или приближение занятия;
2. находятся затронутые группы, преподаватели и периоды;
3. выбираются подписчики;
4. формируется текст уведомления;
5. сообщение отправляется через внешний канал;
6. формируется отчёт об отправке.

Текущий канал доставки — Telegram. Отправщик учитывает ограничения Telegram API,
повторяет отправку при временных ошибках и помечает заблокированные чаты.

## Фоновые Задачи

Celery используется для выполнения асинхронных и периодических операций:

- запуск сбора расписания;
- синхронизация данных;
- отправка уведомлений;
- планирование напоминаний о занятиях;
- резервное копирование базы данных;
- очистка старых данных.

В development-окружении поднимаются:

- `celery-worker` — выполнение фоновых задач;
- `celery-beat` — планировщик периодических задач;
- `flower` — мониторинг Celery.



# Production запуск (через Docker)

Production compose-файл поднимает:

- PostgreSQL;
- Redis;
- Django через Gunicorn;
- Celery worker;
- Celery Beat;
- Flower;
- Nginx;
- Certbot.

Перед запуском необходимо создать и заполнить `.env`.

Если используется внешний Docker network из compose-файла:

```bash
docker network create backend_net
```

Запуск:

```bash
docker compose up -d --build
```

Миграции:

```bash
docker compose exec django python manage.py migrate
```

Сбор статических файлов:

```bash
docker compose exec django python manage.py collectstatic --noinput
```

## Полезные Команды

```bash
python manage.py migrate
python manage.py createsuperuser
python manage.py filltemplate
python manage.py collectstatic --noinput
```

Команда `filltemplate` создаёт стандартный шаблон времени занятий, который
используется при формировании учебных периодов.

## Примечания

- Не храните реальные секреты в репозитории.
- Для сопровождения данных используйте Django Admin вместо ручного изменения БД.
- Перед синхронизацией с базой сырые данные расписания должны быть подготовлены
  в ожидаемом промежуточном формате.
